# 爬虫链路全面测试与修复记录 (2026-04-20)

## 测试目标

删除所有 API skill（index.js + adapter.js），通过心跳触发 recovery 链路，让 Agent 从零开始：爬虫拦截 API → 生成脚本 → 执行验证 → data-store 入库。验证整个自维护闭环是否能跑通。

## 测试结果

全链路跑通。携程公网、Trip.com、美团后台、携程后台均成功拦截 API、生成脚本、入库。

## 发现并修复的 Bug

### Bug 1：CDP 锁残留导致爬虫卡死

**现象**：爬虫日志停在 `ENTERING BACKGROUND MODE` 后无任何输出，300 秒后被 safety timeout 杀掉。

**根因**：`browser-controller.ts` 的 CDP 锁（`hotel-ai-browser-cdp.lock`）设计为进程退出时通过 `process.once('exit')` 释放。但 `skill-executor.ts` 用 `SIGKILL` 杀超时进程时，Node.js 的 `exit` 事件不会触发，锁文件残留。下一个爬虫等锁最长 10 分钟。

**修复（两层防护）**：
1. `src/main/skill-executor.ts`：子进程 `close` 事件中，如果是 `ai-web-crawler`，读取锁文件 PID 与子进程 PID 对比，匹配则删除锁文件
2. `scripts/ai-web-crawler/src/browser-controller.ts`：stale 检测增加 PID 存活性检查（`process.kill(pid, 0)`），死进程立即判定 stale；mtime 阈值从 10 分钟降到 2 分钟

**文件变更**：
- `src/main/skill-executor.ts` — `child.on('close')` 回调增加 CDP 锁清理
- `scripts/ai-web-crawler/src/browser-controller.ts` — stale 检测逻辑重写

---

### Bug 2：Recovery/autoFix 成功后不采集竞品

**现象**：Trip.com 通过 recovery 生成脚本后，只拿到本店 1 条数据，竞品 2 条数据缺失。携程通过直接 API 调用成功，竞品数据正常。

**根因**：`heartbeat-manager.ts` 的 `executeTask` 中，`executeCompetitorTasks()` 只在 try 块的直接 API 成功路径调用。catch 块中 recovery/autoFix 成功后直接 return，没有调竞品采集。

**修复**：catch 块中 `autoFixScript` 成功和 `executeAgentRecovery` 成功后，都补上 `executeCompetitorTasks(task)` 调用。

**文件变更**：
- `src/main/heartbeat/heartbeat-manager.ts` — catch 块增加竞品采集

---

### Bug 3：爬虫 page 匹配到应用自身页面 → 登录误检测 → 布局崩溃

**现象**：登录横幅弹出后，左侧面板极窄，右侧聊天面板占满屏幕。关闭横幅后不恢复。

**根因链**：
1. Trip 爬虫的 bgTab 创建后，`context.on('page')` 事件超时（新 BrowserView 可能属于新 context）
2. domain fallback 也失败（`trip.com` 被重定向到 `us.trip.com`）
3. 兜底扫描匹配到了 `file:///...dist/renderer/index.html`（应用自身的渲染进程页面）
4. 应用 HTML 里有 `input[type="password"]`、登录按钮、`#loginForm` → 登录检测 confidence=0.8 → 判定为登录页
5. `promoteBgTabToFront` 把 BrowserView 加到 mainWindow 并设置了 `currentBounds` → 遮挡/干扰布局

**修复（三层）**：
1. `browser-controller.ts` — bgTab 创建前快照所有已有页面 URL，fallback 只在新出现的页面中查找（根治）
2. `login-detector.ts` — `file://` 页面直接返回 `isLoginPage: false`（防御层）
3. `window-manager.ts` — `promoteBgTabToFront` 中新 BrowserView 设置零尺寸 `{x:0,y:0,width:0,height:0}`，不遮挡当前页面

**文件变更**：
- `scripts/ai-web-crawler/src/browser-controller.ts` — 快照 + fallback 逻辑重写
- `scripts/ai-web-crawler/src/login-detector.ts` — file:// 排除
- `src/main/window-manager.ts` — promoteBgTabToFront 零尺寸

---

### Bug 4：bgTab 前置清理 sessionId 不匹配（无效清理）

**现象**：长时间运行后 bgTab 累积，CDP 连接变慢。

**根因**：`heartbeat-manager.ts` 的 `executeAgentRecovery` 前置清理写的 sessionId 是 `recovery-${task.id}`（不带时间戳），但 bgTab 创建时用的是 `recovery-${task.id}-${Date.now()}`（带时间戳）。`destroyBgTabsBySession` 是精确匹配，永远匹配不到。

**修复**：`window-manager.ts` 的 `destroyBgTabsBySession` 改为前缀匹配（`entry.sessionId?.startsWith(sessionId)`）。

**文件变更**：
- `src/main/window-manager.ts` — destroyBgTabsBySession 匹配逻辑

---

### Bug 5：登录检测优化

**修改内容**：
1. 初始检测等待策略从 `domcontentloaded + 500ms` 改为 `domcontentloaded + networkidle(8s)`，增加重定向检测
2. 登录按钮关键词收紧：去掉 `next/继续/下一步/提交/submit`（非登录页常见），保留 `获取验证码/发送验证码`（强登录信号）
3. 用户名检测收紧：去掉 `input[name*="user"]`（太宽泛），输入框阈值 ≤4 → ≤3
4. 验证码选择器保留 `verify/slider`（美团滑块验证码需要）

**文件变更**：
- `scripts/ai-web-crawler/src/crawler-orchestrator.ts` — 初始登录检测等待逻辑
- `scripts/ai-web-crawler/src/login-detector.ts` — 打分规则调整

---

### Bug 6（防御性）：split-layout 异常比例保护

`src/renderer/components/split-layout.ts` 的 `applyLayout` 增加安全检查：右侧面板超过窗口 50% 时自动重置为 80:20 默认比例。

## 已知未修复问题

### 1. 前端价格展示去重取最高价

**现象**：Dashboard 显示的房型价格是同房型多个套餐中的最高价，而非最低价。
**原因**：adapter 正确返回了所有套餐的价格记录，但前端展示层去重时取了最后一条（最高价）。
**影响**：数据展示不准确，但入库数据本身是完整正确的。
**修复方向**：前端 Dashboard 的去重逻辑改为取最低价，或展示价格区间。

### 2. Agent 查竞品实时价格时调爬虫

**现象**：用户问"北京三里屯太古里桔子水晶酒店的实时价格"，Agent 先正确调了 `api-ctrip-public-price`，但随后又调了爬虫（参数错误：operation 字段拼入了描述文本，没传 start_url）。
**原因**：LLM（Gemini Flash）的工具调用决策问题，不是代码 bug。Agent 不清楚 `api-ctrip-public-price` 支持传 `hotelId` 查竞品，也不清楚 `data-store parseOnly` 会读最新文件。
**修复方向**：优化 `api-ctrip-public-price` 的 SKILL.md 说明，明确竞品查询用法。

### 3. Agent 重复执行脚本

**现象**：携程后台 recovery 过程中，Agent 在 2 分钟内执行了 8 次 `api-ctrip-backend-price`。
**原因**：Gemini Flash 倾向于"验证再验证"，不够果断。不影响数据正确性，但浪费 API 调用和时间。
**修复方向**：prompt 层面加更强的约束（"验证成功后立即入库，禁止重复执行"）。

### 4. api-results 文件累积

**现象**：每次 API 调用都会在 `data/api-results/` 下生成新文件，长期运行后文件越来越多。
**修复方向**：定期清理旧文件（保留最近 N 个或最近 N 天）。

### 5. 磁盘空间

测试中出现 `ENOSPC: no space left on device`，需要清理磁盘空间。
