# Trip.com 动态站点适配修复 (2026-04-17)

## 问题背景

Trip.com API skill (`api-trip-public-price`) 持续返回 `htlSpiderActionErrorCode: 203`（反爬拦截），根本原因不是 cookie 过期，而是 **prompt 规则与实际浏览器行为不一致**：

1. 多个 prompt 文件强制要求 Agent 把 Trip.com 域名改成 `www.trip.com`、locale 改成 `zh-CN/CNY/CN`
2. 实际浏览器访问 `www.trip.com` 会重定向到 `us.trip.com`（美国站），拦截到的真实请求域名、locale、currency 都是 `en-US/USD/US`
3. `schedule.json` 中 `cookieDomain` 固定为 `www.trip.com`，而 `CookieService` 只按精确域名取 cookie，无法覆盖 `us.trip.com` 的 cookie
4. 结果：Agent 生成的脚本域名、cookie、浏览器会话三者不一致 → 203

## 修改内容

### 1. Prompt 源头规则修正（3 个文件）

**目标**：删除"强制 `www.trip.com` + `zh-CN/CNY/CN`"的规则，改为"按拦截结果复制真实域名和 locale"。

| 文件 | 修改点 |
|------|--------|
| `agent/prompts/recovery.md:81` | Trip 特殊规则改为"必须复制拦截到的真实域名，不要跨域名硬改" |
| `agent/prompts/recovery-strategy-a.md:61` | 同上 |
| `agent/prompts/script-examples.md` | 示例2 URL 改为 `us.trip.com`；head locale 改为 `en-US/USD`；referer 改为 `us.trip.com`；关键点和通用规则第5条都改为"以拦截结果为准" |

### 2. `tasks/schedule.json` cookieDomain 改为父域

```diff
- "cookieDomain": "www.trip.com"
+ "cookieDomain": "trip.com"
```

**注意**：linter/外部工具会自动恢复 schedule.json 的某些字段（如 cron 时间），需确认 `cookieDomain` 改动被保留。

### 3. `src/main/cookie-service.ts` 父域 cookie 补查

当传入的是父域（如 `trip.com`，只有2段），额外执行 `cookies.get({ domain: '.trip.com' })` 查询，合并所有子域 cookie 后去重返回。

这样 `cookieDomain=trip.com` 时，同时覆盖 `.trip.com`、`us.trip.com`、`www.trip.com` 的会话 cookie。

不影响已有精确子域任务（如 `ebooking.ctrip.com`、`me.meituan.com`），因为它们的域名有3段以上，不触发父域补查。

## 验证结果

### 成功的部分

1. **Prompt 修正生效**：Recovery Agent 生成的新脚本使用了 `us.trip.com` 域名和 `en-US/USD` locale（之前会被 prompt 强制改回 `www.trip.com/zh-CN/CNY`）
2. **API 请求成功**：不再返回 `htlSpiderActionErrorCode: 203`，返回了 `Ack: "Success"` 的正常响应
3. **Adapter 解析正常**：解析出 5 条记录（来自 `compensatedRooms`，因为 `roomList` 当天 soldOut）
4. **data-store 入库成功**

### 已知问题

1. **`roomList` 为空 / `isRoomListSoldOut: true`**：当天该酒店在 Trip.com 上房间全部售罄，只有 `compensatedRooms`（补偿推荐房）有数据。这是业务状态，不是代码问题。
2. **Agent 调用时传了 `cookieDomain: "www.trip.com"`**：Recovery prompt 模板里的 `cookieDomain` 写死了 `www.trip.com`。虽然 CookieService 的父域增强可以兜底，但更好的做法是让 recovery prompt 从 `schedule.json` 动态读取 cookieDomain。
3. **`schedule.json` 被 linter 回写**：外部工具会修改此文件的 cron/时间字段，可能连带回写 `cookieDomain`。需确认 `trip.com` 被保留。
4. **Agent 只复制了 4 个 header**：虽然 prompt 里有"完整复制 requestHeaders"的铁律，但 Agent（Gemini Flash）仍然只保留了 `accept/content-type/referer/user-agent`。Trip 公网目前风控宽松所以没报错，但后续如果 Trip 加强风控可能又会触发 203。这是 Agent 行为问题，不是本次修复的范围。

## 附加修复：icon-debug 日志洪泛

### 问题
爬虫在 Trip.com 等页面上，每个导航步骤会输出 60-70 行 `[icon-debug] Selector: "icon-底部右侧" from position-fallback`，因为页面有大量 fixed 定位元素没有 aria-label/class/svg 可用于识别，全部走 position-fallback 路径，且 debug log 在去重检查之前输出。

### 修复
`scripts/ai-web-crawler/src/smart-click-handler.ts:596`：跳过 `position-fallback` 类型的 debug log（这些元素通过位置生成 selector，重复且无信息量）。

### 影响
- 仅减少 stderr 日志噪音，不影响功能
- 有 aria-label、class、svg-href 的元素仍正常输出 debug log

### 5. Recovery 策略 A/B 命名统一

`recovery.md` 中策略 A/B 的命名与代码（`heartbeat-manager.ts`）及策略子文件（`recovery-strategy-a.md` = 爬虫、`recovery-strategy-b.md` = 本地修复）相反。已重写 `recovery.md`，统一为：

| 标识 | 含义 | 对应文件 |
|------|------|----------|
| 策略A | 爬虫重建（ai-web-crawler） | `recovery-strategy-a.md` |
| 策略B | 本地修复（读写脚本/adapter） | `recovery-strategy-b.md` |

代码中 `needsCrawler ? 'a' : 'b'`、策略子文件内容均未改动，仅 `recovery.md` 的叙述文本做了 A↔B 翻转。
