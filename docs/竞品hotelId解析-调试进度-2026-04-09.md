## 一、背景

在竞品功能的 Phase 2 中，需要为用户添加的竞品酒店解析其在携程/Trip.com 上的 hotelId，以便 Phase 3 复用现有 API 脚本采集竞品价格。

核心 API：`POST https://m.ctrip.com/restapi/soa2/34951/fetchRecommendList`
- 支持关键词搜索（`destination.keyword.word`）和坐标附近搜索（`recommend.nearbyHotHotel.coordinate`）
- 响应路径：`data.recommendHotel.hotelList[].hotelInfo.summary.hotelId`
- pageSize 限制最大 25

## 三、当前卡点：搜索脚本返回空数据

### 3.1 现象

- Agent 成功拦截到 `fetchRecommendList` API
- Agent 成功生成搜索脚本并执行
- API 返回 `Ack: "Success"` 但 `data` 里只有 `ResponseStatus`，没有 `recommendHotel`
- 同样的 payload 在浏览器 Console 里（无论带不带 Cookie、带不带 phantom-token）都能返回 10 条酒店

### 3.2 已排除的原因

| 假设 | 验证方式 | 结论 |
|------|---------|------|
| `aid`/`sid` 为空导致 | 浏览器 Console 用空 `aid`/`sid` 测试 | ❌ 不是原因，返回 10 条 |
| `phantom-token` 过期导致 | 删掉 phantom-token 后测试 | ❌ 不是原因，脚本仍返回空 |
| `phantom-token` 假值导致 | 浏览器 Console 用 `'1004-common-test'` 测试 | ❌ 不是原因，返回 10 条 |
| Cookie 未注入 | 价格脚本 `api-ctrip-public-price` 同样的 Cookie 机制能正常工作 | ❌ Cookie 机制本身没问题 |
| `filters` 缺少 `80|2` 和 `29|1` | 补上后测试 | ❌ 不是原因 |
| 不带 Cookie 就不行 | 浏览器 Console 不带 `credentials: 'include'` 测试 | ❌ 不需要 Cookie 也返回 10 条 |

### 3.3 当前最可能的原因

**Node.js `https.request` 和浏览器 `fetch` 的行为差异**：浏览器 fetch 跨域请求时自动带 `Origin` header，Node.js 不带。携程可能检查 `Origin` header。

**最新修复**：在脚本 headers 中加了 `'origin': 'https://hotels.ctrip.com'`。**待验证。**

### 3.4 如果 Origin 不是原因

下一步排查方向：
1. 在 `api-runtime/index.js` 的 `fetch` 方法里加日志，打印实际发出的完整 headers（包括 Cookie）
2. 对比 Node.js 发出的请求和浏览器发出的请求的所有 header 差异
3. 可能需要加 `user-agent` header（Node.js 默认不带）

## 四、Agent 写脚本的已知问题

Agent 通过爬虫拦截 API 后自动生成脚本时，存在以下反复出现的问题：

1. **pageSize 写成 50**：携程限制最大 25，Agent 每次都需要自己修复（能自动修复，但浪费迭代次数）
2. **`aid`/`sid` 被清空**：Agent 把这些值当成"动态 session ID"而清空，实际上是固定的渠道标识
3. **`phantom-token` 被硬编码**：从拦截结果中复制了过期的 token，应该不带这个 header
4. **requestBody 字段丢失**：即使用了 `JSON.parse` 模式，Agent 仍然会修改它认为"不需要"的字段

这些问题已通过 `api-first-instruction.md` 的改进部分缓解，但 LLM 不会 100% 遵守。长期方案应该在工具层面解决（如爬虫把 requestBody 保存为模板文件，脚本直接读取）。
