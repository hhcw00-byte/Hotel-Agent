## 一、本次已解决的问题

### 1.1 搜索脚本空数据问题 — 已解决

**原问题**：`fetchRecommendList`（soa2/34951）返回 `Ack: "Success"` 但无酒店数据。

**排查过程**：
1. 对比能工作的价格脚本 `api-ctrip-public-price/index.js` 和搜索脚本的差异
2. 发现搜索脚本缺少 `user-agent` header 和 `head` 对象中的 `ctok`/`lang`/`syscode` 等字段
3. 补齐后数据返回了，但返回的全是同品牌推荐（`hotelType: "COMPENSATE"`），不是附近真实酒店

**根因**：`fetchRecommendList` 是推荐 API，不是搜索 API。不管怎么调参数，它返回的都是携程推荐算法的结果。

### 1.2 找到正确的 API — 已解决

**排查过程**：
1. `fetchRecommendList`（soa2/34951）❌ — 推荐 API，返回同品牌推荐
2. `ctGetNearbyHotelList`（soa2/28820）❌ — 详情页附近酒店，只返回 20 家且受算法影响，覆盖率不够
3. `gaHotelSearchEngine`（soa2/21881）✅ — 搜索建议/自动补全 API，按酒店名精确匹配返回 hotelId

**最终方案**：用 `gaHotelSearchEngine`，对每个竞品酒店名调一次，从结果中匹配 `type: "Hotel"` 的条目提取 `id` 作为 hotelId。

**优势**：
- 纯文本匹配，不受推荐算法影响
- 不需要 Cookie/登录态（公开接口）
- 每个竞品单独查，100% 覆盖率
- 在搜索框输入时触发，不需要点搜索按钮

## 四、关键架构决策记录

### 4.1 为什么用 `gaHotelSearchEngine` 而不是其他 API

| API | 问题 |
|-----|------|
| `fetchRecommendList`（soa2/34951） | 推荐 API，返回同品牌推荐而非搜索结果 |
| `ctGetNearbyHotelList`（soa2/28820） | 只返回 20 家，受推荐算法影响，覆盖率不够 |
| `gaHotelSearchEngine`（soa2/21881） | ✅ 搜索建议接口，按酒店名精确匹配，不需要 Cookie，100% 覆盖率 |

### 4.2 "已验证 API 记录"机制

在 `competitor-hotelid-resolver/SKILL.md` 中加了"已验证的 API 记录"区块：
- Agent 每次发现正确/错误的 API 后必须更新此区块
- 下次执行时先查记录，有 ✅ 的直接用，不重复试错
- 这个机制对其他平台（美团、Booking）同样适用

### 4.3 通用搜索建议 API 识别规则

在 `api-first-instruction.md` 中加了不绑定平台的通用识别规则：
- requestBody 有 `keyword`/`query`/`q` 字段
- responseSize 小（< 10KB）
- 返回 `id` + `name` 列表
- 在输入时触发（不是点搜索按钮后）

### 4.4 脚本编写关键要点（Agent 必须遵守）

从本次调试中总结的关键要点，已写入 `api-first-instruction.md`：
1. **必须带 `user-agent` header** — Node.js 默认不发 UA，缺少会导致返回空数据
2. **`head` 对象必须完整保留** — `ctok`、`lang`、`syscode`、`auth`、`xsid` 等字段即使为空也不能删
3. **`aid`/`sid` 必须保留原值** — 这是渠道标识不是 session ID
4. **不要硬编码 `phantom-token`** — 会过期，大部分 API 只靠 Cookie 就能工作
