## 二、各项测试详情

### 2.1 Booking 公网房价采集

**测试方法**：调用 `api-booking-public-price` skill，传入本店 hotelSlug

**结果**：
- HTML 页面正常获取（需要有效 Cookie）
- `b_rooms_available_and_soldout` JSON 数据成功提取
- adapter 正确解析房型名、价格、币种、早餐信息
- data-store 入库成功，platform 标记为 `booking-public`

**指标**：
| 指标 | 结果 |
|------|------|
| 数据获取成功率 | 100%（Cookie 有效时） |
| 房型字段完整性 | roomName、price、currency、date 完整 |
| 价格准确性 | 与页面显示一致（b_raw_price 字段） |

### 2.2 Booking 后台房价采集

**测试方法**：调用 `api-booking-backend-price` skill，传入 hotelId

**结果**：
- ses token 自动从 302 重定向 Location header 中获取，无需手动配置
- GraphQL `roomInventoryQuery` 接口返回未来 30 天房价 + 房态
- 数据结构：rooms[].rates[].dates[].price[{occupancy, value}]

**指标**：
| 指标 | 结果 |
|------|------|
| ses 自动获取 | ✅ 从 302 重定向提取 |
| 数据获取成功率 | 100%（Cookie 有效时） |
| 数据覆盖范围 | 未来 30 天 |
| 房态数据 | roomsToSell、netBooked 字段完整 |

### 2.3 Booking 酒店搜索（slug 解析）

**测试方法**：调用 `api-booking-hotel-search`，传入酒店名关键词

**结果**：
- GraphQL AutoComplete 接口正常返回酒店列表（需要 Cookie）
- 搜索结果页 HTML（~1.7MB）中成功用正则精确匹配 slug
- 品牌名 + 地理位置词双重校验防止错误匹配

**指标**：
| 指标 | 结果 |
|------|------|
| 搜索成功率 | 100%（酒店在 Booking 存在时） |
| 精确匹配率 | 约 70%（部分酒店名称差异大时降级为 NO_MATCH） |
| 错误匹配率 | 0%（品牌校验后不再出现跨品牌错误匹配） |

**已知问题**：
- 部分酒店在 Booking 上的名称与国内名称差异较大（如"桔子水晶"在 Booking 上可能叫"Crystal Orange"），导致搜索无结果
- 品牌列表为硬编码，新品牌需要手动添加

### 2.4 竞品采集链路

**测试方法**：心跳触发 `api-booking-public` 任务后，自动展开竞品采集

**结果**：
- `executeCompetitorTasks` 正确识别 booking 平台，传 `hotelSlug` 参数
- 其他平台（ctrip、trip、meituan）行为不变，传 `hotelId`
- 单个竞品失败不影响其他竞品

**指标**：
| 指标 | 结果 |
|------|------|
| 平台隔离性 | ✅ booking 传 slug，其他传 hotelId |
| 容错性 | ✅ 单条失败不阻塞 |
| 入库正确性 | ✅ hotel_id 正确关联 competitors 表 |

**指标**：

## 四、后续优化方向

1. **酒店名称匹配**：考虑加入英文名匹配（Booking 上部分酒店用英文名）
2. **异常提示优化**：Cookie 过期、网络超时等场景的用户友好提示
3. **界面易用性**：需要进一步用户测试和迭代
4. **酒店搜索配置**：在管理后台加入酒店搜索功能，简化 hotelId 配置流程
5. **Booking 后台 ses 稳定性**：监控 ses 过期频率，评估是否需要自动刷新机制
