## 一、背景与目标

### 1.1 当前系统能力

系统已实现**本店数据采集闭环**：

```
HeartbeatManager 定时触发
  → 执行 API 脚本（scripts/api-xxx/index.js）
  → 获取携程/美团/Trip.com/Booking 的房价和房态
  → data-store + adapter 清洗入库（MySQL price_snapshots / room_snapshots）
  → Dashboard 展示（hotel-admin.html 数据看板）
  → 脚本失败时 Agent Recovery 自动修复
```

### 1.2 缺失的能力

**没有竞品数据**。酒店需要知道周边竞品的定价策略，才能制定自身的定价决策。

### 1.3 目标

1. 用户能从地图/列表中选择周边竞品酒店
2. 系统自动获取竞品在各平台的 hotelId
3. 复用现有 API 脚本（传入不同 hotelId）定时采集竞品价格
4. Dashboard 展示本店 vs 竞品价格对比

## 四、Phase 2 设计：竞品 hotelId 解析（方案已确定）

### 4.1 问题

用户选了竞品酒店（有名称和坐标），但要从各平台采集该竞品的价格，需要知道它在各平台的 hotelId。

### 4.2 已验证的关键结论

#### hotelId 可替换 — 已验证通过

现有 API 脚本已参数化 hotelId：

```javascript
// scripts/api-ctrip-public-price/index.js:12
const hotelId = params.hotelId || 129175148;

// scripts/api-trip-public-price/index.js:10
const hotelId = params.hotelId || 129175148;
```

测试结果（2026-04-09）：
- 将 hotelId 改为 `133059446`（另一家酒店）→ **成功获取到该酒店的房价数据**
- phantom-token 是 session 级别（不绑定特定 hotelId），过期由 Agent Recovery 自动刷新

**结论：只要拿到竞品的 hotelId，直接复用现有脚本，传入 `{ hotelId: xxx }` 参数即可采集竞品价格。**

#### 携程搜索 API — 已找到 `fetchRecommendList`

**端点**：`POST https://m.ctrip.com/restapi/soa2/34951/fetchRecommendList`

该 API 是携程酒店列表页的核心搜索接口，支持：
- **坐标附近搜索**：通过 `recommend.nearbyHotHotel.coordinate` 传入本店经纬度
- **关键词搜索**：通过 `destination.keyword.word` 传入酒店名
- **分页**：`paging.pageIndex / pageSize`，单次最多可返回较多结果

响应结构：
```
data.recommendHotel.hotelList[].hotelInfo.summary.hotelId      → "894450"
data.recommendHotel.hotelList[].hotelInfo.nameInfo.name         → "汉庭酒店(北京长虹桥团结湖地铁站店)"
data.recommendHotel.hotelList[].hotelInfo.positionInfo.address  → "农展馆南里12号院5号楼"
data.recommendHotel.hotelList[].hotelInfo.positionInfo.mapCoordinate[0].latitude/longitude
data.recommendHotel.hotelList[].hotelInfo.commentInfo.commentScore → "4.7"
data.recommendHotel.hotelList[].roomInfo[0].priceInfo.price     → 295（最低价）
```

**关键发现**：`hotelTotalCount: 366`，说明一次坐标搜索可覆盖附近所有酒店。

### 4.3 确定方案：纯 HTTP 批量解析（零爬虫）

#### 核心流程

```
hotel_config 取本店 hotelId + 坐标
  ↓
调 fetchRecommendList（坐标附近模式，pageSize=50）
  ↓
提取 hotelList 中所有 { hotelId, hotelName, address, lat, lng }
  ↓
与 competitors 表做名称模糊匹配
  ↓
匹配成功 → 写入 competitor_platform_ids (platform='ctrip')
  ↓
未匹配的竞品 → 用酒店名做关键词搜索兜底（单独请求）
  ↓
携程 hotelId 同时适用于 Trip.com → 写入 platform='trip'
```

#### 名称匹配算法

高德 POI 名称和携程酒店名通常高度一致，但可能有细微差异：
- 高德：`汉庭酒店(北京长虹桥团结湖地铁站店)`
- 携程：`汉庭酒店(北京长虹桥团结湖地铁站店)` （通常一致）

匹配策略：
1. **精确匹配**：去除空格和标点后完全一致
2. **核心词匹配**：提取品牌名+门店名关键词（如"汉庭"+"长虹桥"），两者都命中即匹配
3. **坐标距离兜底**：名称匹配不上时，用坐标距离 < 200m 作为辅助判断

#### 实现方式

新建脚本 `scripts/api-ctrip-hotel-search/index.js`：

```javascript
// 输入参数
params.lat        // 本店纬度（从 hotel_config）
params.lng        // 本店经度
params.hotelId    // 本店携程 hotelId
params.cityId     // 城市ID（默认1=北京）
params.keyword    // 可选：关键词搜索模式
params.pageSize   // 可选：默认50

// 输出
{
  success: true,
  data: {
    hotels: [
      { hotelId: "894450", name: "汉庭酒店(...)", address: "...", lat: 39.939, lng: 116.469, score: "4.7", minPrice: 295 },
      ...
    ],
    totalCount: 366,
    pageIndex: 1
  }
}
```

#### 触发时机

- **自动触发**：用户在竞品配置页添加竞品后，系统自动调用解析
- **手动触发**：竞品列表页增加"解析平台ID"按钮
- **批量解析**：一次坐标搜索覆盖大部分竞品，未匹配的逐个关键词搜索
- **Agent 驱动**：通过 `competitor-hotelid-resolver` strategy skill 指导 Agent 完成全流程

#### Agent 自动化流程（核心）

新建 strategy skill `skills/competitor-hotelid-resolver/SKILL.md`，指导 Agent：

```
Step 1: 检查是否已有 api-ctrip-hotel-search 脚本
  ↓ 没有
Step 2: 爬虫导航到 hotels.ctrip.com → 搜索本店名 → 拦截 fetchRecommendList API
Step 3: Agent 分析 apiCandidates → 写 scripts/api-ctrip-hotel-search/index.js + adapter.js
  ↓ 已有
Step 4: 执行搜索脚本（坐标附近模式）→ 获取酒店列表 → 和 competitors 表做名称匹配
Step 5: 匹配成功 → 写入 competitor_platform_ids（ctrip + trip 两条记录）
Step 6: 未匹配的 → 用酒店名关键词搜索兜底
```

**关键设计**：脚本由 Agent 通过爬虫 API 发现后自动生成，不需要人工编写。
脚本失效时走标准 Recovery 流程自动修复，实现自维护。

#### 各平台 hotelId 解析策略

| 平台 | 方案 | 说明 |
|------|------|------|
| 携程 | fetchRecommendList API | ✅ 已确认可行，纯 HTTP |
| Trip.com | 复用携程 hotelId | ✅ 已验证共享同一 hotelId |
| 美团 | 待发现类似搜索 API | 美团后台可能有竞品搜索接口 |
| Booking | 待研究 | URL slug 模式，可能需要爬虫 |

**优先级**：先完成携程（覆盖携程+Trip.com），其他平台后续迭代。

## 五、Phase 3 设计：竞品数据采集（方案已设计）

### 5.1 前置修复（Phase 3 开始前必须完成）

#### 问题 1：`insertPriceSnapshots` 未传入 hotel_id

当前 `insertPriceSnapshots()` 的 INSERT 语句中没有 `hotel_id` 字段，所有数据都写成 `hotel_id=0`（本店）。

**修复**：`insertPriceSnapshots` 和 `insertRoomSnapshots` 增加可选 `hotelId` 参数。

#### 问题 2：`data-store` 无 hotel_id 透传链路

当前调用链：
```
HeartbeatManager → executeSkill('data-store', { source: 'api-ctrip-public-price' })
  → data-store/index.js → insertPriceSnapshots(platform, records)  // 无 hotelId
```

**修复**：data-store 接受 `hotelId` 参数并透传给 `insertPriceSnapshots`：
```
HeartbeatManager → executeSkill('data-store', { source: 'api-ctrip-public-price', hotelId: 123 })
  → data-store/index.js → insertPriceSnapshots(platform, records, hotelId)
```

### 5.2 采集架构：HeartbeatManager 动态展开竞品任务

**不在 schedule.json 里写竞品任务**，而是在执行本店任务时动态展开：

```
executeTask('api-ctrip-public') — 本店采集（hotelId=默认）
  ↓ 成功后
查询 competitor_platform_ids WHERE platform='ctrip' AND platform_hotel_id IS NOT NULL
  ↓
为每个竞品执行同一个 skill：
  executeSkill('api-ctrip-public-price', { hotelId: 竞品ctripId })
  executeSkill('data-store', { source: 'api-ctrip-public-price', hotelId: competitors.id })
```

**选择动态展开而非静态配置的原因**：
- 竞品是动态的（用户随时增删），绑定到静态 schedule.json 会造成数据不一致
- 增删竞品只改数据库，不需要动 schedule.json
- 竞品采集频率可以独立控制（低于本店频率）

### 5.3 竞品采集频率

竞品采集应低于本店频率，避免请求量膨胀：
- 本店：每 2 小时（或按 schedule.json 配置）
- 竞品：每 6 小时或每天 2 次
- 可在 HeartbeatTask 中增加 `competitorCron` 字段，或在展开逻辑中用独立计时器

### 5.4 数据存储

`price_snapshots` 表已添加 `hotel_id` 列：
- `hotel_id = 0`：本店数据（现有数据默认值）
- `hotel_id > 0`：竞品数据（FK → competitors.id）

### 5.5 Dashboard 竞品对比（待设计）

在数据展示面板增加"竞品对比"Tab，展示：
- 本店 vs 各竞品的价格列表对比（按日期维度）
- **Phase 3 先不做跨酒店房型对齐**，直接按竞品原始房型名展示
- 后续再用 LLM 做智能房型对齐（本店"豪华大床房" ↔ 竞品"高级大床房"）
