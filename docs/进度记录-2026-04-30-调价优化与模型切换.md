## 二、全平台 stdout 污染修复

### 2.1 问题
`console.log` 在 Node.js 中写入 stdout，和 JSON 结果混在一起，导致父进程 `extractTrailingJson` 解析失败。Agent 收到解析错误后误报"调价失败"，实际调价已成功。

### 2.2 修复
- **ctrip/trip**: 所有 `console.log` → `console.error`（用 Node.js 替换，避免 PowerShell 编码问题）
- **meituan**: 7 处 `console.log` → `console.error`
- **booking**: 已在 bgTab 集成时处理

### 2.3 教训
**不要用 PowerShell 的 `Get-Content | Set-Content` 处理 UTF-8 文件**，会破坏中文编码。用 Node.js 的 `fs.readFileSync/writeFileSync` 替代。

## 六、登录检测重构

### 6.1 统一为 LoginDetector bgTab 方式
- **文件**: `src/main/skill-manager.ts`
- **之前**: ctrip/trip 用 API 探测（读系统 Chrome cookie），meituan/booking 用 LoginDetector
- **之后**: ctrip/trip/meituan 统一用 LoginDetector bgTab（通过 Electron 内嵌浏览器 session 检测）
- **原因**: API 探测依赖系统 Chrome cookie 数据库，Electron 环境下经常读取失败

### 6.2 Booking 跳过预检
- Booking 从 `PLATFORM_LOGIN_MAP` 移除，不做登录预检
- **原因**: Booking 的 WAF（awswaf.com）对频繁打开后台页面敏感，登录预检本身会触发安全机制导致 session 失效
- Booking 的登录态由 verify-script.js 内部检测（URL 重定向到 sign-in）

### 6.3 LoginDetector 增强
- **文件**: `scripts/login-check/index.js`
- 修复 `finalUrl` 获取：等待 `load` 事件，URL 为空时遍历所有 page 找重定向后的页面
- 新增 `domIsLogin` 判定：`hasPasswordInput + hasLoginForm` 同时为 true 直接判定为登录页

### 6.4 当前登录检测方式

| 平台 | 检测方式 | 检测 URL |
|------|----------|----------|
| ctrip | login-check bgTab + LoginDetector | ebooking.ctrip.com |
| trip | login-check bgTab + LoginDetector | ebooking.trip.com |
| meituan | login-check bgTab + LoginDetector | me.meituan.com |
| booking | 不预检（verify-script 内部检测） | — |

## 八、模型自动切换

### 8.1 问题
主模型（OpenRouter）在中国不可用，返回 `403 This model is not available in your region`。前端直接报错，没有切换到备用模型。

### 8.2 根因
- `streamCallOpenAIAPI`（流式调用）没有 fallback 逻辑
- `callOpenAIAPI`（非流式调用）有 fallback，但用户对话走的是流式
- `llm-config.json` 中没有配置 `fallback` 字段

### 8.3 修复

#### 代码修改

**`src/main/pi-agent-manager.ts`**:
```javascript
// 流式调用添加 fallback
const stream = await this.openaiClient.chat.completions.create(streamParams)
  .catch(async (err) => {
    // 主模型报错，立即切换到备用模型重试
    if (!isUsingFallback() && this.reinitializeWithFallback()) {
      streamParams.model = this.config.model;
      return this.openaiClient.chat.completions.create(streamParams);
    }
    throw err;
  });
```

**`src/shared/llm-config-loader.ts`**:
- `isProviderUnavailableError` 扩展错误匹配：
  - 新增错误码：`UND_ERR_SOCKET`, `ERR_TLS_CERT_ALTNAME_INVALID`
  - 新增消息匹配：`socket hang up`, `ECONNRESET`, `ENOTFOUND`, `getaddrinfo`, `ssl`, `tls`, `abort`
  - 新增 OpenRouter 特有错误：`openrouter`, `rate.limit`, `quota`, `billing`, `blocked`, `unavailable`
  - 新增错误类型匹配：`ConnectionError`, `NetworkError`, `FetchError`, `AbortError`, `TypeError`

#### 配置修改

**`llm-config.json`**:
```json
{
  "provider": "openai",
  "apiKey": "<openrouter-key>",
  "baseURL": "https://openrouter.ai/api/v1",
  "model": "google/gemini-3.1-flash-lite-preview",
  "temperature": 0.7,
  "maxTokens": 4096,
  "fallback": {
    "apiKey": "<novaiapi-key>",
    "baseURL": "https://us.novaiapi.com/v1",
    "model": "gemini-3-flash-preview"
  }
}
```

### 8.4 切换流程

```
用户发送消息
  → streamCallOpenAIAPI 调用主模型（OpenRouter）
  → 主模型返回 403 "not available in your region"
  → .catch() 捕获错误
  → isProviderUnavailableError() 判断为 provider 不可用
  → reinitializeWithFallback() 切换到备用模型
    → switchToFallback() 设置 _usingFallback = true
    → loadLLMConfig() 返回 fallback 配置
    → 重新创建 OpenAI client（新 apiKey + baseURL）
  → 用备用模型（novaiapi）重试请求
  → 成功返回响应
```

### 8.5 注意事项
- `llm-config.json` 首次加载后会自动迁移到加密的 `llm-config.enc`，明文文件被删除
- 修改配置需要：先删除 `llm-config.enc`（项目根目录 + `%APPDATA%/hotel-ai-browser/`），再创建 `llm-config.json`，重启应用
- 切换到 fallback 后，当前进程生命周期内一直使用 fallback，重启应用后恢复主模型
- 非流式调用（tool calling 场景）的 fallback 逻辑之前已有，本次只修复了流式调用
