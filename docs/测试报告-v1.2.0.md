## 二、各项测试详情

### 2.1 多轮对话能力

连续10轮对话测试，Agent 能正确维持上下文，不出现话题丢失或逻辑断裂。具体表现：
- 跨轮次引用前文信息准确
- 复杂业务场景（查价格 → 对比竞品 → 给调价建议）链路完整
- 对话历史从数据库加载，重启后上下文不丢失

### 2.2 Agent 幻觉测试

Agent 在无数据时明确告知"暂无数据"，不编造房价、房态等业务数据。具体表现：
- 数据库无记录时返回空结果说明，不虚构数字
- 调用 API skill 失败时如实报告错误，不伪造成功结果
- 竞品信息严格来源于数据库 `competitors` 表，不凭空生成竞品名称

### 2.3 爬虫稳定性（待优化）

15次爬虫任务执行，11次成功，4次失败。

**成功场景**：
- PMS 房态数据采集稳定，导航到房态预测页后 API 拦截成功率高
- Trip.com 公网价格采集稳定，酒店详情页 API 拦截正常
- ctrip.com 公网价格采集稳定，酒店详情页 API 拦截正常
- 搜索 API 拦截（竞品 hotelId 解析）流程可用

**失败原因分析**：
- 携程公网价反爬检测（`htlSpiderActionErrorCode: 203`），脚本中硬编码的 `cid`/`vid` 过期或缺少反爬 header
- 多爬虫并发时 CDP 连接偶发超时（已通过信号量限制并发数为2缓解，但未完全消除）

**后续优化方向**：

- 对比浏览器真实请求与脚本请求的 header 差异，补全反爬字段
- 增加 CDP 连接重试机制和超时容错

### 2.4 数据准确性

采集数据与实际网站数据对比，各字段验证结果：

| 字段 | 验证结果 |
|------|----------|
| 房型名称 | 完整匹配，通过 adapter 标准化 |
| 价格 | ctrip 精确匹配，支持多日期价格日历，trip币种换算有些偏差待优化 |
| 房态（可售房量） | 与 PMS 实时数据一致 |
| 日期 | 动态计算，不存在硬编码过期问题 |
| 竞品信息 | hotelId 与平台实际 ID 一致 |

### 2.5 自修复能力（待优化）

10次 API 失效场景触发自修复，6次成功恢复数据采集。成功率 60%。

**失败场景**：
- Agent 遇到反爬拦截时倾向于直接重新调爬虫，而非先读脚本排查问题
- 携程 203 错误码属于平台级反爬，脚本层面无法绕过，需要 Cookie/Header 级别的修复
- 偶发情况下 Agent 生成的脚本不符合 APIRuntime 规范（使用了 `axios`、`https.request` 等禁止写法）

**后续优化方向**：
- 在 `recovery.md` 中强化"先读脚本诊断 → 再决定是否重新拦截"的策略优先级
- 增加脚本规范校验（生成后自动检查是否使用了 APIRuntime）
- 针对反爬错误码建立专门的修复策略（Cookie 刷新、Header 补全）

### 2.6 闭环验证

心跳任务执行完成后的闭环验证机制工作正常，10次调用：
- `HeartbeatManager.executeTask()` 直接执行 API 脚本，不经过 Agent
- 脚本返回结果经 `isEmptyAPIResponse()` 检测数据是否为空壳
- 空壳数据触发 `executeAgentRecovery()`，Recovery 完成后 `verifyRecoveryData()` 独立验证
- 数据文件决定成败，不依赖 Agent 的文字回复
